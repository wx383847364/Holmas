# 校验与回归策略

## 校验分层

### 1. 输入校验

校验 `DesignPacket` 是否缺少：

- 页面标识
- 设计图
- 状态定义
- 必要标注

当前 input 层只负责产出未决项评估，不直接替代 `UiPrefabSpec`。

当前最小 input 闭环要求：

- intake analyzer 能区分 blocking issue 和 requires human decision
- blocking issue 存在时不得直接进入正式 spec 解释
- sample design packet 能稳定解释成当前 sample spec

### 2. Spec 校验

校验 `UiPrefabSpec` 是否存在：

- 空 `page_id`
- 空 `prefab_name`
- 缺失根节点
- 重复节点 ID
- 无效父子关系
- 非法组件类型

### 3. 输出校验

校验 `PrefabBindingManifest` 是否存在：

- 缺失节点路径
- 缺失组件类型
- 非法组件类型
- 资源组件缺 `asset_slot`
- 重复 entry 或命名冲突
- 需要人工接线却未标记
- 生成路径不在 profile 允许目录内

## 回归要求

- 同一 spec 重复生成，结果必须稳定
- 修改单个局部节点，不允许全 prefab 无关漂移
- 校验结果必须可脚本化执行
- 回归输出应能区分 schema、generator、adapter、fixture 问题

## Validation Baseline

`Subagent 5` 全面介入前，至少具备：

- 1 份 approved sample spec
- 1 份 sample manifest
- 1 份 Holmas adapter 样例
- 可复跑的最小 golden case

当前最小 baseline 已冻结为：

- `sample_ui_prefab_spec.json`
- `sample_prefab_binding_manifest.json`
- `sample_holmas_generated_result_plan.json`
- `validation_baseline.json`

第一版 baseline 重点覆盖：

- sample design packet -> sample spec
- sample spec -> sample manifest
- sample manifest -> sample Holmas generated result plan
- design packet interpretation
- manifest 输出校验
- Holmas generated result 消费校验
- deterministic manifest regression

## 第一版最小验收

- layout 检查脚本可验证专区目录与关键文件存在
- schema 相关测试可跑通
- Holmas adapter 只提供项目映射，不依赖 gameplay 业务代码
- validator 能前置拦截非法节点、非法组件、缺资源位和命名冲突

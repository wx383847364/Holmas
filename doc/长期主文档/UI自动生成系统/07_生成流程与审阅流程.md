# 生成流程与审阅流程

## 固定流水线

1. 设计侧提交 `DesignPacket`
2. `Subagent 2` 输出 `UiPrefabSpec`
3. 人工审阅 spec
4. `Subagent 3` 生成 prefab 草稿和 `PrefabBindingManifest`
5. `Subagent 4` 补 Holmas profile、目录映射和消费约束
6. `Subagent 5` 跑校验与回归，产出 `validation baseline`
7. `Subagent 6` 给阶段性结论
8. Holmas UI 实现方接业务逻辑

## 启动门

- `Subagent 1` 先交付 `contracts freeze note`
- `Subagent 5` 只有在至少有 1 份 approved sample spec 和 1 份 sample manifest 后才全面介入
- `Subagent 6` 只在阶段里程碑后介入，不对每个中间结果做同步硬门

## 审阅点

### 审 spec

重点看：

- 页面树是否合理
- 节点命名是否稳定
- 资源位和绑定位是否齐全
- 不确定项是否显式暴露

### 审 prefab 草稿

重点看：

- 节点层级是否符合 spec
- 组件和布局是否符合 profile
- binding manifest 是否可用于后续接线

## 明确禁止

- 禁止从图片直接跳到正式 prefab
- 禁止未审 spec 就生成正式业务资产
- 禁止把 spec、生成、验证混在同一批文件里
- 禁止在 Holmas 业务 UI 目录里堆生成器实现

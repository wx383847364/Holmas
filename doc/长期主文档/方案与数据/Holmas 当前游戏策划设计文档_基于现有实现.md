# Holmas 当前游戏策划设计文档（基于现有实现）

## 文档目的

本文档用于把 Holmas 当前仓库里的设计文档、导出配置与运行时代码收敛成一份可核对版本，方便系统设计、程序实现与策划口径对齐。

本文档以当前仓库中的以下事实为准：

- 正式导出配置：`Assets/Config/json/holmas_core_config.json`
- 正式猫配置：`Assets/Config/json/holmas_cat_meta.json`
- 运行时主链：`HolmasGameplayRuntime`、`HolmasLevelLaunchGateway`、`HolmasTaskProgressService`、`HolmasAgencyProgressionService`
- 当前自动化验证口径：`Assets/Tests/HolmasConfigExportTests.cs`

如果本文档与旧方案文档冲突，以“当前实现”章节标注的结论优先。

## 一、产品定位

Holmas 当前版本是一款“异形扫雷底板上的找猫驱动成长游戏”。

核心体验由四条链路组成：

1. 玩家在主界面接任务。
2. 系统根据当前任务栏生成本局地图的猫种池。
3. 玩家在棋盘中找出全部隐藏猫，推进任务进度。
4. 玩家领取任务金币，用于升级城市宣传功能，宣传升级再推动玩家等级与内容开放。

它不是传统扫雷胜负规则，也不是纯收集挂机，而是：

- 棋盘负责“找猫与点击体验”
- 任务负责“中期目标”
- 宣传系统负责“长期成长”
- 离线收益负责“保底经济循环”

## 二、当前版本范围

当前仓库已经实现并接通的系统有：

- 正式配置加载与二进制恢复
- 本地档案读写与恢复未完成关卡
- 任务栏生成、领奖、广告槽位时效
- 基于任务猫池的地图生成
- 棋盘找猫、数字提示、自动扩散、通关判定
- 宣传升级、经验增长、阶段推进
- 离线金币收益
- 体力消耗与自然恢复
- Main 内嵌找猫主流程

当前明确不在正式闭环内的内容有：

- 博彩任务
- 第二货币
- 宣传效果数值产出
- 复杂战斗或角色养成系统
- 真服务器，当前仍以本地模拟档案为主

## 三、核心玩法闭环

### 3.1 单局循环

1. 进入主界面后，系统先补齐当前已解锁任务槽。
2. 玩家点击“开始找猫/继续找猫”。
3. 系统读取当前任务栏中的唯一猫种，生成本局 `CatPool`。
4. 系统按玩家等级可用地图池抽取一张地图模板，并按地图配置生成本局猫总数。
5. 猫只会从当前任务猫池中按权重生成到有效格。
6. 玩家在棋盘中点击格子，逐步找出全部猫。
7. 找出本局全部猫后，关卡完成，任务按猫种累计进度。
8. 已完成任务可领取金币奖励，并立即补入新任务。
9. 玩家用金币升级当前城市宣传项，获得经验并可能推进到下一城市阶段。
10. 更高等级和更高城市阶段继续开放更强地图、任务与成长空间。

### 3.2 长期循环

长期循环不是“打图直接涨等级”，而是：

`找猫 -> 完成任务 -> 获得金币 -> 升级宣传 -> 获得经验 -> 提升等级 -> 解锁更高难内容`

这意味着：

- 任务和离线收益提供金币
- 宣传升级是经验的核心来源
- 地图完成本身主要承担任务推进与猫收集功能

## 四、系统设计拆解

### 4.1 任务系统

任务栏后端固定为 `5` 个槽位：

- 默认开启 `2` 个
- 广告临时解锁 `3` 个

任务生成规则：

- 按玩家等级配置中的 `TaskTypeIds + TaskTypeWeights` 抽任务模板
- 每个任务模板再从自己的 `CatIdList` 中抽 1 个猫种
- 当前任务栏内 `CatId` 不允许重复
- 如果当前等级下没有足够的唯一猫种，允许任务栏补不满，不强制重复

任务实例字段包括：

- `taskInstanceId`
- `sourceTaskTypeId`
- `taskKind`
- `catId`
- `targetCount`
- `currentCount`
- `reward`
- `slotIndex`
- `expireAt`

当前仅支持 `Money` 任务，奖励公式为：

`reward = cat.price * targetCount * levelRewardFactor`

任务推进规则：

- 关卡完成后，统计本局出现的每种猫数量
- 任务按相同 `catId` 累加进度
- 达到 `targetCount` 后变为可领奖

任务领奖规则：

- 只能领取已完成未领取任务
- 领奖只增加金币
- 领奖后槽位清空
- 若槽位仍处于解锁状态，立即补新任务

广告槽位规则：

- 解锁后立即补任务
- 到期后若槽位内仍有任务，直接清空并重新锁定
- 到期时间由当前玩家等级配置中的 `AdUnlockHours` 决定

### 4.2 地图与关卡系统

地图来源于 `MinesweeperTerrainData` 对应的地形模板，当前地图配置只负责：

- `mapId`
- `terrainPath`
- `catCountMin`
- `catCountMax`

地图请求生成规则：

- 先按玩家等级配置中的 `MapIds + MapWeights` 抽地图
- 再将当前任务栏中活跃任务的唯一猫种转成 `CatPool`
- `CatPool` 中每个猫种权重来自猫表 `Weight`

关卡快照内容包括：

- 地图 ID
- 地形路径
- 随机种子
- 本局生成的猫列表
- 已翻开格记录
- 完成状态

地图生成边界：

- 猫只会生成在有效格
- 同一格只放一只猫
- 猫总数会被有效格数量上限截断
- 如果当前任务栏没有可用猫池，则当前不能启动地图

### 4.3 棋盘玩法规则

棋盘当前采用“找出所有猫即通关”的规则，不是传统扫雷的“排除所有雷”。

基础规则：

- 点到猫格会直接翻出该猫
- 点到非猫且周围数字为 `0` 的格子会自动扩散
- 点到有数字的普通格只翻当前格
- 可以插旗，但当前主流程点击翻格会忽略旗阻挡

完成条件：

- 当本局所有生成猫都被找出后，关卡立即完成

当前还会展示：

- 棋盘尺寸
- 本局总猫数 / 已找到猫数
- 剩余隐藏猫数
- 当前任务进度摘要

### 4.4 体力系统

当前已接入正式运行时，规则如下：

- 默认上限 `50`
- 默认初始体力 `50`
- 每 `12` 分钟自然恢复 `1`
- 当前仓库还有一个 `+5 体力` 的调试/验证入口

两种交互模式：

- `Walk`
  - 翻到非猫格不扣体力
  - 翻到猫格扣 `2`
- `Find`
  - 翻任何有效且未翻开的格子固定扣 `1`

体力不足时：

- 不允许执行本次翻格
- 返回失败提示“体力不足”

### 4.5 经济系统

当前只有单一主要货币：金币。

金币来源：

- 任务领奖
- 离线结算

金币消耗：

- 升级当前城市阶段下的宣传功能

当前没有正式接入的内容：

- 任务外直接掉落金币
- 第二货币
- 宣传升级以外的大额消耗口

### 4.6 城市宣传成长系统

当前实现已经把“建筑成长”主语义切到“城市宣传成长”。

阶段结构：

- 当前配置有 `100` 个城市阶段 `AgencyStageId`
- 每个阶段固定 `4` 个宣传项
  - `leaflet`
  - `radio`
  - `online`
  - `tv`
- 每个宣传项固定上限 `5` 级

升级规则：

1. 检查当前阶段是否存在该宣传项。
2. 检查是否已满级。
3. 检查该级费用是否合法且金币是否足够。
4. 扣除金币。
5. 对应宣传等级 `+1`。
6. 玩家经验固定 `+1`。
7. 重算玩家等级。
8. 若当前阶段四个宣传项全部满级，则推进到下一城市阶段。

经验规则：

- 宣传每升 1 次，玩家经验固定 `+1`
- 地图完成不直接加经验
- 任务领奖不直接加经验
- 离线结算不直接加经验

阶段推进规则：

- 必须当前阶段 4 项宣传全部满级才推进
- 如果已经是最后阶段，则不再越界推进

### 4.7 玩家等级系统

当前正式导出配置中，玩家等级使用 `PlayerLevels` 驱动。

每级主要承载：

- `UpgradeExp`
- `OfflineRewardPerHour`
- `AdUnlockHours`
- `TaskTypeIds + TaskTypeWeights`
- `MapIds + MapWeights`

玩家等级的作用：

- 决定可抽哪些任务模板
- 决定可抽哪些地图
- 决定离线收益时薪
- 决定广告槽位解锁时长

当前实现中，玩家等级重算依据是累计经验是否达到下一等级的 `UpgradeExp` 门槛。

### 4.8 离线收益系统

离线收益当前只提供金币。

计算方式：

- 按当前玩家等级对应的 `OfflineRewardPerHour`
- 按离线毫秒数线性折算

离线结算会：

- 增加金币余额
- 累计到 `OfflineRewardTotal`
- 记录最近一次离线结算时间

不会：

- 直接增加经验
- 直接推进任务
- 直接推进宣传等级

### 4.9 存档系统

当前以本地模拟服务器档案作为 authoritative archive。

档案内容包括：

- 玩家长期成长数据
- 任务栏状态
- 当前未完成关卡
- 宣传等级
- 体力状态
- 猫发现计数

启动恢复时：

- 先加载本地档案
- 若档案损坏，回退默认新号
- 若任务栏不一致，重建任务栏并保留其余长期进度
- 若存在未完成关卡，则尝试恢复该关卡

## 六、当前实现与旧方案文档差异

这一节是系统核对时最需要关注的内容。

### 6.1 玩家等级数量已不是旧文档里的 20 级

旧方案文档中曾写：

- 玩家等级 `20`
- 总经验池 `2000`

但当前正式导出配置与测试口径已经是：

- 玩家等级 `100`
- `PlayerLevels` 共 `100` 行
- 当前配置最后一级 `UpgradeExp = 1476`

因此如果你现在在核对系统方案，玩家等级设计应以“100 级正式配置”视角核对，而不是旧的 20 级方案。

### 6.2 宣传系统已是正式主语义

旧文档里还保留过“建筑成长”的迁移表述，但当前运行时与测试已经以：

- `Promotion`
- `AgencyStage`
- `TryUpgradePromotion`

作为正式口径。

系统设计层面不建议再把 `building` 作为主语义继续扩展。

### 6.3 Main 当前只显示前 4 个任务槽

后端任务栏容量仍是 `5`，但当前 `MainPresenter` 里界面只取前 `4` 个槽位展示。

这意味着：

- 系统层：5 槽位成立
- 当前主界面表现层：只展示 4 槽位

如果你正在核对 UI 系统设计，这里是一个需要特别确认的实现差异点。

### 6.4 找猫主入口已切到 Main 内嵌棋盘

近期迭代记录显示，当前主要体验入口已不是单独 Battle 页面跳转，而是：

- Main 页面内嵌棋盘
- 支持继续当前未完成关卡
- 找完后可继续推进下一局

如果系统方案里仍按“主界面 -> 独立 Battle 页”作为唯一正路径，需要更新。

### 6.5 体力系统已纳入正式运行时

旧玩法主文档对体力描述不足，但当前代码已存在完整规则：

- 上限
- 自然恢复
- Walk/Find 双模式
- 体力不足拦截
- 存档持久化

因此体力系统已经不是临时调试项，而是正式需要被系统方案承接的玩法模块。

## 七、建议核对清单

你在核对系统设计方案时，可以优先逐项确认：

1. 玩家等级是按 `100` 级正式配置做的，还是还停留在旧 `20` 级方案。
2. 任务栏系统是否明确区分“5 槽后端能力”和“当前 Main 仅展示 4 槽”的表现差异。
3. 地图生成是否明确依赖“当前任务栏唯一猫池”，而不是独立随机猫池。
4. 棋盘通关条件是否定义为“找到全部猫”，而不是传统扫雷规则。
5. 经验来源是否锁定为“宣传升级成功事件”，而不是任务领奖/过图/挂机。
6. 宣传系统是否按“100 城市阶段 x 4 宣传项 x 5 级”建模。
7. 体力系统是否已进入正式系统边界，包括 Walk/Find 两套扣费规则。
8. 存档方案是否覆盖未完成关卡恢复、任务栏恢复、体力恢复与宣传等级恢复。
9. 主流程页面是否按“Main 内嵌找猫”为当前正式体验路径来设计。
10. 自动化验证口径是否与正式配置一致，尤其是地图池、等级池和宣传阶段数量。

## 八、结论

Holmas 当前版本已经形成一条完整的正式主链：

`任务驱动地图 -> 地图驱动找猫 -> 找猫推进任务 -> 任务产出金币 -> 金币升级宣传 -> 宣传产出经验 -> 等级开放更高内容`

从系统设计角度看，当前最重要的不是再补抽象概念，而是统一以下三套口径：

- 旧方案文档里的目标口径
- 当前导出配置的正式数值口径
- 当前运行时代码的真实执行口径

如果后续要继续扩设计文档，建议直接以本文档为“当前版本策划基线”，再往上拆成：

- 数值策划文档
- 任务与地图产出文档
- 宣传成长文档
- UI 表现与交互文档
- 存档与恢复规则文档

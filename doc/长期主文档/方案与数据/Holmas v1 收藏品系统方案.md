# Holmas v1 收藏品系统方案
**Summary**

- 本方案用于为 Holmas 设计首期收藏品系统，目标是在现有找猫玩法之上增加一层长期收集驱动。
- 收藏品的首期获得来源固定为两类：
  - 完成找猫任务后的奖励
  - 活动奖励
- 收藏品系统首期按“静态配置表 + 本地存档 + 运行时奖励投放”落地，不要求同步接入真实服务端。
- 收藏品既服务于长期留存，也服务于活动运营，因此静态定义与奖励来源配置需要分层，不直接把收藏品写死在任务逻辑里。
- 收藏品首期不采用“获得本体即解锁”，而是采用“收集碎片后解锁完整收藏品”的模式。

**Goals**

- 给 Holmas 增加一套可扩展的收藏品收集框架，支持后续继续接入签到、节日活动、兑换商店等来源。
- 让玩家在完成核心找猫循环后，持续获得“收集新物件”的长期反馈，而不只获得金币。
- 保持收藏品定义、奖励来源、存档进度和 UI 展示之间的边界清晰，方便后续做新系列和新活动。
- 让任务奖励和活动奖励都能复用同一套收藏品发放协议，避免出现两套库存和两套记录逻辑。

**Non-Goals**

- 首期不做玩家间交易、赠送、拍卖行。
- 首期不做收藏品数值养成，不给收藏品增加升级、强化、装备位。
- 首期不做收藏品拆解、兑换、合成。
- 首期不做服务端背包校验，只以本地存档为准。
- 首期不做复杂图鉴故事系统，先保证收集、展示、来源和奖励链路可用。

**Collectible Definitions**

- 收藏品定义为“可被玩家永久收集并在图鉴中展示的非消耗型内容”。
- 每个收藏品至少包含以下基础属性：
  - `collectibleId`
  - `displayName`
  - `seriesId`
  - `rarity`
  - `iconPath`
  - `description`
  - `sortOrder`
  - `isEnabled`
- 首期建议收藏品按“系列”组织，例如：
  - 城市纪念物
  - 猫咪周边
  - 节日限定纪念章
- `rarity` 只用于展示和掉落配置，不直接提供玩法加成。

**Acquisition Rules**

- 收藏品的获得来源首期固定为两类：
  - `TaskReward`
  - `ActivityReward`
- `TaskReward`
  - 只允许从“完成找猫任务后的奖励发放阶段”进入收藏品系统
  - 不从翻格、找到猫瞬间直接掉落
  - 这样可以保证奖励反馈与任务完成闭环一致
- `ActivityReward`
  - 只允许从活动结算、活动里程碑、活动邮件或活动兑换阶段发放
  - 不与常驻任务奖励口径混用
- 首期推荐两种发放模式：
  - 直接获得指定收藏品碎片
  - 从某个收藏品碎片奖励池中抽取一个碎片奖励
- 收藏品解锁规则固定为：
  - 玩家先获得某个 `collectibleId` 对应的碎片
  - 当累计碎片数达到解锁需求后，收藏品从“未解锁”变为“已解锁”
  - 解锁后继续获得相同碎片时，碎片数量继续累计，不丢失

**Duplicate Rules**

- 收藏品的实际收集单位首期改为“碎片”。
- 同一 `collectibleId` 的碎片会持续累计，无论该收藏品是否已经解锁。
- 当碎片数首次达到 `unlockShardCount` 时：
  - 该收藏品记为“已解锁”
  - 记录首次解锁时间
- 解锁后继续获得相同碎片时：
  - 不再重复触发“首次解锁”
  - 只累计额外碎片数量
- 首期不再使用 `duplicateCount` 作为主状态，后续若要做碎片兑换，可直接基于累计碎片扩展。

**Reward Protocol**

- 建议统一新增收藏品奖励项协议：
  - `rewardType = CollectibleShard`
  - `rewardValue = collectibleId`
  - `rewardCount = shardCount`
- 若要支持奖池抽取，则增加奖励组定义：
  - `rewardType = CollectibleShardPool`
  - `rewardValue = collectiblePoolId`
  - `rewardCount = shardCount`
- 任务奖励链路：
  - 任务完成
  - 结算常规金币奖励
  - 再结算可选收藏品碎片奖励
  - 若本次碎片累计后刚好解锁新收藏品，则弹出“新收藏品解锁”反馈
- 活动奖励链路：
  - 活动达成条件
  - 读取活动奖励表
  - 若奖励项包含收藏品碎片，则走同一套碎片库存落账逻辑

**Data Model**

- 建议在 Shared / 配置层固定以下模型概念：
  - `CollectibleDefinition`
  - `CollectibleSeriesDefinition`
  - `CollectibleRewardPoolDefinition`
  - `CollectibleGrantResult`
- `CollectibleGrantResult` 至少包含：
  - `collectibleId`
  - `isNew`
  - `grantedShardCount`
  - `ownedShardCountAfterGrant`
  - `isUnlockedAfterGrant`
  - `sourceType`
  - `sourceId`

**Config Plan**

- 首期建议至少新增两张配置表：
  - `Holmas_CollectibleTable.xlsx`
  - `Holmas_CollectibleRewardPoolTable.xlsx`
- `Holmas_CollectibleTable.xlsx` 建议字段：
  - `collectibleId`
  - `displayName`
  - `seriesId`
  - `rarity`
  - `iconPath`
  - `description`
  - `unlockShardCount`
  - `sortOrder`
  - `isEnabled`
  - `notes`
- `Holmas_CollectibleRewardPoolTable.xlsx` 建议字段：
  - `poolId`
  - `poolName`
  - `collectibleIds`
  - `weights`
  - `shardCounts`
  - `notes`
- 任务表与活动表不直接复制收藏品静态属性，只引用：
  - `collectibleId`
  - 或 `collectiblePoolId`

**Archive Changes**

- 扩展玩家长期存档，新增收藏品进度结构：
  - `UnlockedCollectibleIds`
  - `CollectibleShardCounts`
  - `CollectibleFirstUnlockedAtUtcMilliseconds`
  - `CollectibleLastUpdatedAtUtcMilliseconds`
- 若后续活动需要按系列统计，还可追加：
  - `UnlockedCollectibleSeriesIds`
  - 但首期可以运行时从 `OwnedCollectibleIds` 推导，不强制落档

**Runtime Architecture**

- 建议在 `App.HotUpdate` 新增收藏品运行时服务，例如：
  - `HolmasCollectibleService`
- 其职责固定为：
  - 根据 `collectibleId` 或 `collectiblePoolId` 发放收藏品碎片
  - 判断碎片累计后是否首次解锁
  - 写入玩家存档
  - 产出统一的 UI 通知结果
- 任务系统和活动系统都不直接操作存档字段，只调用收藏品服务。

**Task Integration**

- 找猫任务奖励扩展建议：
  - 常规任务默认仍以金币为主
  - 部分任务可额外挂一个收藏品碎片奖励项
  - 高价值任务可挂收藏品碎片奖池而非固定碎片项
- 这样做的好处是：
  - 不破坏现有任务主循环
  - 可以逐步给少量任务试点收藏品掉落
  - 更适合后续活动周或主题周临时加投放

**Activity Integration**

- 活动系统接入建议：
  - 活动里程碑奖励支持直接引用 `collectibleId + shardCount`
  - 活动兑换或盲盒奖励支持引用 `collectiblePoolId`
- 活动奖励与常驻任务奖励共用同一套收藏品碎片库存，不开第二套活动背包。
- 对于活动限定收藏品，建议在 `seriesId` 或 `notes` 中明确标识活动归属，方便图鉴筛选和后续复刻。

**UI Plan**

- 首期建议新增“收藏品 / 图鉴”页面，入口可放在 `Main` 页。
- 页面结构建议分为三块：
  - 系列筛选区
  - 收藏品网格列表
  - 详情面板
- 收藏品展示界面首期建议补齐以下功能：
  - 图鉴总进度展示
  - 当前系列进度展示
  - 稀有度与来源标签展示
  - 新获得标记展示
  - 当前碎片进度展示
  - 未解锁占位展示
  - 来源说明与获得记录展示
- 页面顶部建议固定展示：
  - `已解锁数量 / 总数量`
  - 当前选中系列名称
  - 当前系列完成度，例如 `8 / 20`
- 系列筛选区建议支持：
  - 查看全部收藏品
  - 按 `seriesId` 切换不同系列
  - 显示每个系列的已解锁数量与总数量
- 列表区建议支持以下筛选与排序能力：
  - 按“全部 / 已解锁 / 未解锁”筛选
  - 按稀有度筛选
  - 按来源筛选，例如“任务奖励 / 活动奖励”
  - 按默认配置顺序排序
  - 按最近获得时间排序
- 收藏品网格列表建议补齐以下表现：
  - 已解锁状态显示正常彩色图标
  - 未解锁状态显示灰态图标或问号占位
  - 新获得但尚未查看的收藏品显示 `New` 标记
  - 未解锁时显示碎片进度，例如 `12/30`
  - 已解锁后可继续显示累计碎片数，例如 `38`
- 每个收藏品格子至少展示：
  - 图标
  - 稀有度
  - 是否已解锁
  - 当前碎片数量 / 解锁所需碎片
- 点击某个收藏品后，详情面板建议展示：
  - 大图标
  - 名称
  - 稀有度
  - 所属系列
  - 文案描述
  - 获得来源说明
  - 首次获得时间
  - 当前碎片数量
  - 解锁所需碎片数量
  - 当前解锁状态
- 对未解锁收藏品，详情面板建议展示：
  - 基础轮廓信息，例如系列、稀有度
  - 简要提示“可通过任务奖励或活动奖励获得”
  - 明确显示还差多少碎片可解锁
  - 不强制展示完整描述，避免完全剧透
- 如果收藏品是活动限定，建议额外展示：
  - `活动限定` 标签
  - 活动归属说明
  - 后续是否可能复刻的文案占位字段
- 首次获得时建议弹出单独的获得浮层：
  - 显示收藏品图标
  - 显示名称
  - 显示来源，例如“任务奖励”或“活动奖励”
- 如果本次获得的是碎片，建议浮层文案区分两类：
  - 仅获得碎片但未解锁：强调“收藏品碎片 +N”
  - 本次累计后完成解锁：强调“新收藏品解锁”
- 建议在收藏品页内增加“未读清理”机制：
  - 玩家进入收藏品页并查看后，清掉对应 `New` 标记
  - 不要求点击单个卡片后才清理，也可以按进入页面批量清理
- 未解锁收藏品首期可以显示灰态占位，不强制隐藏。
- 空状态建议：
- 新号首次进入时显示“还没有获得任何收藏品”
  - 同时提示主要获得途径为任务奖励碎片和活动奖励碎片
- 交互细节建议：
  - 进入页面默认定位到“全部”
  - 若玩家刚通过活动获得收藏品，可优先定位到对应系列
  - 切换系列或筛选项时保留当前滚动位置策略需要提前定死，首期建议切换后回到顶部

**Progression Value**

- 收藏品本身首期不提供直接战斗或收益加成。
- 它的价值主要来自：
  - 图鉴完成度
  - 系列收集目标
  - 活动限定稀缺性
  - 长期回流动机
- 若后续要加入图鉴总进度奖励，可以作为第二阶段扩展，不在本方案首期范围内绑定。

**Failure Handling**

- 如果奖励结算时收藏品定义缺失：
  - 记录错误日志
  - 跳过该收藏品碎片项
  - 不影响同批次金币等奖励发放
- 如果奖池配置非法：
  - 不进行抽取
  - 记录错误日志
  - 任务或活动主体流程继续
- 如果存档中存在已移除的旧收藏品 ID：
  - 首期不主动清档
  - 图鉴页面可忽略找不到定义的条目

**Test Plan**

- 新号初始进入图鉴页：
  - 没有任何已解锁收藏品
  - 未解锁格子状态正常显示
- 完成带收藏品奖励的找猫任务：
  - 金币奖励正常发放
  - 收藏品碎片正确入库
  - 未达到阈值时不解锁
  - 达到阈值时 `isNew=true`
- 重复完成同一奖励任务：
  - 不重复创建第二份收藏品定义
  - 只增加碎片数量
- 活动奖励发放指定收藏品：
  - 若碎片累计未达阈值，则保持未解锁
  - 若碎片累计达到阈值，则正常解锁
  - 若已解锁，则继续正确累计碎片
- 奖池奖励发放：
  - 能按权重返回合法收藏品碎片配置
  - 不会返回不存在的 `collectibleId`
- 存档恢复后：
  - 图鉴已解锁状态保持一致
  - 碎片数量保持一致

**Implementation Order**

- 第一步：冻结收藏品领域模型与存档结构
- 第二步：新增收藏品静态配置表与奖池表
- 第三步：实现 `HolmasCollectibleService`
- 第四步：把找猫任务奖励接入收藏品发放协议
- 第五步：把活动奖励接入同一发放协议
- 第六步：补图鉴 UI 与获得反馈

**Assumptions**

- 首期收藏品只做“收集碎片、解锁并展示”，不做强化、交易和兑换。
- 找猫任务奖励指的是“任务完成后的奖励结算”，不是翻格瞬间掉落。
- 活动系统后续会存在可配置奖励入口，因此本方案只定义收藏品奖励接入边界，不反向重做活动系统主体。
- 解锁后的额外碎片先继续累计，不立刻转化为新货币。

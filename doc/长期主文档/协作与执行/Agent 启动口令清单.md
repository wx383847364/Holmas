# Agent 启动口令清单

## Summary

这份清单只保留最常用、可直接复制的启动口令。  
如果你不想每次都自己组织语言，就直接从这里复制一句发给我。

详细规则看：

- [Agent 启动与验收规范](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Agent 启动与验收规范.md)

## 先看总数

当前项目默认先用 **6 个 Agent**。

1. `Agent 1：边界与骨架`
2. `Agent 2：地图与棋盘`
3. `Agent 3：任务与长期进度`
4. `Agent 4：UI 与验证`
5. `Agent 5：测试与质量保障`
6. `Agent 6：挑刺与问题审查`

如果后续协作更细，再扩到长期版 **9 个 Agent**。
但如果你没有特别说明，我都会先按这 6 个来理解。

## 先看该用哪个 Agent

1. `Agent 1：边界与骨架`
- 适合：冻结 DTO、补 HotUpdate 骨架、守住 Shared 和入口边界
- 你可以理解成：先打地基的 agent

2. `Agent 2：地图与棋盘`
- 适合：地图生成、棋盘结构、terrain 转换、揭示扩散、数字计算
- 你可以理解成：做地图和扫雷核心玩法的 agent

3. `Agent 3：任务与长期进度`
- 适合：任务栏、任务抽取、奖励、领奖补位、长期成长、广告/离线接入口
- 你可以理解成：做任务和成长系统的 agent

4. `Agent 4：UI 与验证`
- 适合：任务栏界面、领奖界面、主界面、结算、UI 绑定、联调验证
- 你可以理解成：做界面接线和冒烟验证的 agent

5. `Agent 5：测试与质量保障`
- 适合：单元测试、集成测试、校验脚本、回归测试、验证其他 agent 的实现
- 你可以理解成：专门做测试和质量把关的 agent

6. `Agent 6：挑刺与问题审查`
- 适合：独立挑刺、找问题、做阻塞性审查、给退回意见
- 你可以理解成：专门在阶段里程碑完成后找问题和卡验收门，并持续负责同一条复审链的 agent

## 最常用判断

- 要先定边界、定 DTO、定入口：用 `Agent 1`
- 要做地图、棋盘、terrain、扫雷规则：用 `Agent 2`
- 要做任务、奖励、任务推进、长期成长：用 `Agent 3`
- 要做界面、按钮、绑定、流程联调：用 `Agent 4`
- 要做单元测试、回归测试、验证别人实现是否正确：用 `Agent 5`
- 要在任务完成后独立挑刺、找问题、决定是否退回修复：用 `Agent 6`

## 最短启动口令

以后你最少只要复制这句，再补一个目标就够了：

```text
按长期主文档里的 Agent 启动与验收规范，执行 Agent 1。
目标：……
```

这句属于默认轻量模式：

- 只表示主线程按对应 Agent 职责做事
- 不默认启动真实 subagent
- 不默认开启“Agent 6 退回后自动补位”的闭环

## 通用模板

```text
按长期主文档里的 Agent 启动与验收规范，执行 Agent X。
目标：……
是否启动真实 subagent：是 / 否。
完成后按验收规范汇报，并执行文档维护流程。
```

如果这轮要用 helper role 先做只读判断或规则审计，统一先看：

- [线程级辅助 subagent 角色](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/线程级辅助 subagent 角色.md)

固定原则：

- 先查线程里有没有同职责 helper
- 有就优先复用，不重复新开
- 只有任务类型变了、写入边界变了、旧实例不再适合复用，或 `context_compression_count >= 2` 时才新开

如果你希望这轮默认按真实 subagent 自动闭环执行，直接用下面这句短口令：

```text
按长期主文档规则执行，默认启动subagent。
```

它固定等价于“默认启动真实 subagent，并开启默认真实 subagent 自动闭环”。

```text

目标：为当前阶段补单元测试、集成测试和回归验证，并检查其他 agent 的实现是否正确。
这次不要主改业务实现。
完成后按验收规范汇报，并执行文档维护流程。
```

## Agent 6

```text
按长期主文档里的 Agent 启动与验收规范，执行 Agent 6。
目标：对当前交付做独立挑刺与问题审查，给出 通过 / 通过，但有非阻塞建议 / 未通过，退回修复 的结论。
这次不要主改业务实现，只输出问题、归属和复审条件。
完成后按验收规范汇报，并执行文档维护流程。
```

## Agent 6 复审

```text
优先让上一轮同一审查链的 Agent 6 继续复审；只有原实例超时、不可用或上下文明显失真时，才改由同职责 reviewer 接手同一 review_chain_id。
目标：基于上一轮同一审查链给出的未通过结论，检查这轮修复是否已经满足复审条件，并给出 通过 / 通过，但有非阻塞建议 / 未通过，退回修复 的结论。
这次仍然不要主改业务实现，只输出剩余问题、严重级别、修复归属和新的复审条件。
完成后按验收规范汇报，并执行文档维护流程。
```

## 启动真实 subagent

如果你要我真的去开一个 subagent，而不是让我自己按某个 agent 规范做事，就把这句写清楚：

```text
按长期主文档里的 Agent 启动与验收规范，启动 Agent 2 真实 subagent。
目标：……
完成后把修改文件、输入输出、风险和阻塞汇报给我，并执行文档维护流程。
```

```text
目标：只读查看长期主文档、迭代记录和必要代码入口，输出当前主线、当前阻塞、下一步建议，以及建议从哪个执行型 Agent 继续。
```

```text
按长期主文档规则执行。

目标：只读审计协作规则、收尾规则、文档维护规则和 subagent 复用规则，输出规则缺口、误判点和最值得补的规则项。
```

## 默认真实 subagent 自动闭环

如果你希望这轮不只是“启动真实 subagent”，还要在 Agent 6 退回后自动继续闭环，最短就说：

```text
按长期主文档规则执行，默认启动subagent。
```

完整展开等价于：

```text
按长期主文档里的 Agent 启动与验收规范执行。
这轮默认启动真实 subagent，并开启默认真实 subagent 自动闭环。
每条实现链达到阶段里程碑后，都按 实现线基础验收 -> Agent 6 审查 -> 原实现线修复 -> 同一审查链复审 -> 通过后再宣告该里程碑完成 的顺序单独闭环。
如果 Agent 6 退回 findings 且当前没有原实现真实 subagent，就自动补起同职责真实 subagent 继续修，不要停下来等我再次下指令。
完成后按验收规范汇报，并执行文档维护流程。
```

## Agent 6 退回后自动补位

如果你已经拿到了 Agent 6 findings，希望我立即继续往下推进，可以直接这样说：

```text
按长期主文档里的 Agent 启动与验收规范继续当前修复链。
Agent 6 已经退回 findings。
先检查当前有没有对应职责的原实现真实 subagent。
如果有，就退回原实现方继续修。
如果没有，而当前线程已开启默认真实 subagent 自动闭环，就自动补起同职责真实 subagent 接手修复。
修完后默认优先交回同一审查链复审；如果原审查实例超时、不可用或上下文明显失真，再由同职责 reviewer 接手同一 review_chain_id。不要停下来等我再次确认。
```

## 未授权自动闭环

如果你这轮只想听结论，不想让我自动补起新的真实 subagent，可以直接这样说：

```text
这轮不要开启默认真实 subagent 自动闭环。
如果 Agent 6 退回 findings，只回显问题和归属，不要自动补起新的真实 subagent。
需要继续修复时我再单独指定。
```

## 顺序控制口令

如果你不想一次开多个 agent，可以直接这样说：

```text
先按规范执行 Agent 2。
等 Agent 2 达到阶段里程碑并完成基础验收后，先交给 Agent 6 做挑刺审查。
Agent 5 可以提前搭测试骨架，也可以在需要时再补专项验证。
如果 Agent 6 不通过，就先退回 Agent 2 修复；修完后默认优先交还同一审查链复审；只有原审查实例超时、不可用或上下文明显失真时，才允许同职责 reviewer 接手。
这轮先不要启动 UI 线。
```

如果你想明确要求“每个 subagent 都要各自过审后，主线程才做最后汇总”，可以直接这样说：

```text
这轮按长期主文档里的严格闭环流程执行：
每个 subagent 都按 达到阶段里程碑 -> Agent 6 审查 -> 原 subagent 修复 -> 同一审查链复审 -> 通过后再宣告该里程碑完成 的顺序单独闭环。
多个 subagent 并行时，各自闭环，不要把 A 的问题交给 B 或主线程顺手修。
主线程只负责分工、收集审查结果、跨 subagent 冲突整合，并在全部通过后再做最终汇总。
除非我明确授权，否则主线程不要代替原 subagent 修 Agent 6 findings。
```

## Assumptions

- 默认“执行 Agent X”表示让我按该 Agent 规范做事，不代表自动创建真实 subagent。
- 只有你明确说“启动真实 subagent”、“启动 subagent”、`默认启动subagent`，或“开启默认真实 subagent 自动闭环”，我才会真的开 subagent。
- 只有你明确开启“默认真实 subagent 自动闭环”，我才会在 Agent 6 退回后静默自动补起同职责真实 subagent。
- 如果你没有特别说明，默认我会按长期主文档中的 Agent 规范和验收规则执行。
- 如果你明确提到“之前那个 Agent 6”或“原来的 Agent 6”，我默认优先复用同一审查链的原审查实例；只有原实例超时、不可用或上下文明显失真时，才允许同职责 reviewer 接手同一 `review_chain_id`。

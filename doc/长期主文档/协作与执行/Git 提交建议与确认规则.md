## Summary

这份文档固定 Holmas 项目的 Git 提交建议与确认规则。

目标固定为：

- 让 `Git 提交建议` 的对外格式保持一致
- 让确认词和 `commit / push` 的执行语义保持一致
- 避免“文档是新规则，工具提示还是旧规则”的口径漂移

## 适用范围、会话锚点与触发时机

这份文档负责：

- `Git 提交建议：适合提交 / 暂不建议提交` 的对外口径
- `标题：`、`内容：` 的展示格式
- 普通提交与 merge 合并提交的标题正文口径
- `提交确认：` 的固定提示词
- 确认词对应的执行语义
- `.git/codex/pending_commit_suggestion.json` 的复用规则
- 模块编号登记文件与提交前编号校验口径

这份文档不负责：

- 收尾触发时机
- `finalize_task.sh`、`check-last-finalize`
- final 里必须显式包含的固定三段

会话锚点规则固定为：

- `@git提交规则` 只对当前会话生效
- 它不是全局默认规则
- 它不替代 [任务完成后自动维护文档](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/任务完成后自动维护文档.md) 的收尾规范
- 只要当前会话已经读到本页，就固定视为当前会话已激活 `@git提交规则`
- 只要本页一旦被读到，主线程就必须立刻结合当前仓库状态，输出一次 `Git 提交建议`
- 不要求用户再额外补一句“请按八位编号方案给出标题和内容”
- 不允许只回复“已读取”“后续按这份规则执行”“知道了”而不继续给出当前仓库的 `Git 提交建议`

固定说明：

- 当前没有专门自动提交脚本，`commit / push` 仍由主线程按确认词执行
- 当前模块最新编号登记在 [commit_module_sequences.json](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/状态登记/commit_module_sequences.json)
- `Git 提交建议` 生成时会先尽量 fetch 远端基线，再结合 git history 与本地登记文件重算下一个编号
- 真正执行 `git commit` 时，`.githooks/commit-msg` 会再次校验标题编号
- 提交成功后，`.githooks/post-commit` 会同步模块编号登记文件，并用 `amend` 把登记更新并入同一个提交

## 读到本页后的立即动作

只要当前会话第一次读到本页，主线程在同一轮回复里固定执行下面动作：

1. 读取当前仓库状态
- 至少检查当前是否在 Git 仓库中、`git status` 是否存在待提交改动、改动边界是否清晰

2. 判断本轮应输出哪一种 `Git 提交建议`
- 如果当前没有待提交改动，直接输出 `Git 提交建议：暂不建议提交。原因是：当前仓库没有待提交改动。`
- 如果当前有待提交改动且边界清晰，直接输出 `Git 提交建议：适合提交`
- 如果当前有待提交改动但边界混杂、验证未完成或明显不应混提，直接输出 `Git 提交建议：暂不建议提交。原因是：xxxx`

3. 如果结论是 `适合提交`
- 必须继续联读 [Holmas 八位提交编号方案](/Users/bruce/work/Holmas/doc/长期主文档/方案与数据/Holmas%20八位提交编号方案.md) 或执行等价编号逻辑
- 必须在同一轮里补齐真实 `标题：`
- 必须在同一轮里补齐 `内容：`
- 必须在同一轮里补齐 `提交确认：...`

4. 如果结论是 `暂不建议提交`
- 也必须在同一轮里明确写出原因
- 如果原因是“当前无待提交改动”，要直接写清楚，不要只说“暂无建议”
- 如果原因是“边界混杂”，要明确指出是哪些改动不应混提，避免用户再追问一次

固定禁止：

- 不允许把“读文档确认”单独作为一次完整回复
- 不允许先只说“规则已读取”，再等用户第二次追问才补 `Git 提交建议`
- 不允许在已经读到本页的同一轮里省略 `标题：`、`内容：` 或 `提交确认：`
- 不允许在已经能判断当前仓库状态的前提下，把“是否需要给提交建议”再次回问给用户

## 联读硬规则

只要当前会话已经读到本页，并且本轮需要输出 `Git 提交建议` 里的 `标题：`，就固定视为：

- 必须继续读取 [Holmas 八位提交编号方案](/Users/bruce/work/Holmas/doc/长期主文档/方案与数据/Holmas%20八位提交编号方案.md)，或执行等价的脚本编号逻辑
- 不允许把 `[TMMSSSSS]`、`[230xxxxx]`、`UI：xxxx` 这类占位写法直接当成最终 `标题：`
- 不允许把“要不要继续看八位编号方案”再次回问给用户

只有下面情况才允许说明阻塞，而不是直接给出真实编号标题：

- 当前目录不在 Git 仓库中
- 当前仓库历史不可读，无法判断该模块下一个编号
- 当前改动边界明显混杂，暂时不能形成单独提交建议

如果当前工作区已有本轮相关改动，且边界足够清晰：

- 主线程应直接结合当前 `git history` 和八位编号方案，给出真实的推荐 `标题：`
- 主线程应直接补完整 `内容：`
- 主线程应直接追加 `提交确认：...`
- 主线程应在读到本页后的第一次有效回复里就把这三项补齐
- 不要先输出占位标题，再让用户补“按八位编号方案改一下”

## Git 提交建议输出格式

只要当前改动已经形成边界清晰、验证完成、适合独立提交的里程碑，就写：

````text
Git 提交建议：适合提交
标题：xxxx
内容：
```text
- xxxx
- xxxx
```
提交确认：如需我继续执行，请回复 1（提交并推送） / 2（只提交） / 确认 / 提交 / 直接提交（只提交）。
````

如果当前改动仍混有多条主线、验证没跑完、审查没完成，或只是事务性协助，就写：

```text
Git 提交建议：暂不建议提交。原因是：xxxx
提交确认：当前不建议直接提交；如需强制提交，请明确说明。
```

如果当前仓库没有待提交改动，也按 `暂不建议提交` 输出，固定写法建议为：

```text
Git 提交建议：暂不建议提交。原因是：当前仓库没有待提交改动。
提交确认：当前不建议直接提交；如需强制提交，请明确说明。
```

固定补充规则：

- 即使这轮只是事务性协助，也必须给出 `Git 提交建议`
- `内容` 统一放进一个 `text` 代码块
- 如果当前规则和旧文档、旧提示词冲突，统一以这份文档和最新工具提示为准
- 如果当前工作区已有 repo 改动，且这轮又要求按本页给提交建议，默认就要继续补到真实编号标题，不要停在占位写法
- 如果当前会话只是“读了本页”，但仓库里已经有待提交改动，也视为已经触发一次完整 `Git 提交建议` 输出义务

## 标题与编号来源

`标题：` 和 `内容：` 默认由 `tools/doc_maintenance/finalize_task.sh` / `tools/doc_maintenance/update_project_docs.py` 生成。

固定要求：

- 主线程可以人工改写脚本生成的 `标题` 与 `内容`
- 但不能省略 `标题：`、`内容：`、`提交确认：`
- 提交标题继续沿用正式八位编号方案
- 读到本页后，只要需要给出 `标题：`，就必须继续联读八位编号方案，或复用脚本已经算出的真实编号结果

这里仅保留最小规则：

- 标题格式固定为 `[TMMSSSSS] 前缀：摘要`
- 所有新建提交的最终标题都必须以 `[TMMSSSSS]` 八位编号开头
- 只有走这套正式提交流程的编号标题才视为占号
- 新建 merge 合并提交也必须走编号标题和正文规则，不能保留 `Merge branch ...` 这类默认英文标题
- 历史上已经存在的旧 merge / 旧无编号提交不回填、不占号；今后不允许新增无编号提交
- `Revert`、`fixup!`、`squash!` 不再绕过编号规则；如需表达这些语义，也必须改写成正式编号标题
- `[TMMSSSSS]` 只是格式说明，不是允许直接对外输出的占位模板

## Commit-msg 强制校验规则

`.githooks/commit-msg` 是最终提交标题闸门。

固定要求：

- 空提交标题直接失败
- 非 `[八位数字] ` 开头的提交标题直接失败
- 已编号标题继续按模块流水号校验，必须是该模块的正确下一个编号
- `commit-msg` 只负责校验标题，不再修改或暂存 `commit_module_sequences.json`
- 默认 `Merge ...` 标题会先按下方 merge 规则自动改写，再进入同一套强制编号校验

## Post-commit 编号登记规则

`.githooks/post-commit` 负责提交成功后的编号登记落盘。

固定要求：

- 根据刚生成的 `HEAD` 和完整 history 同步 `commit_module_sequences.json`
- 如果登记文件发生变化，只把该登记文件 `amend` 进刚刚生成的同一个提交
- 使用递归保护，避免 `post-commit` amend 后再次无限触发自己
- 不在 `commit-msg` 阶段改 index，避免登记文件残留到下一次提交

失败提示固定表达为：

```text
提交标题编号校验失败：
- 当前标题：xxxx
- 原因：提交标题必须以 [TMMSSSSS] 八位编号开头。
```

## Merge 合并提交标题规则

今后只要产生新的 merge 合并提交，标题和正文也必须按本页规则处理。

默认禁止直接使用：

```text
Merge branch 'main' of github.com:xxx/Holmas
Merge remote-tracking branch 'origin/main'
```

固定改成：

```text
标题：[TMMSSSSS] 合并：同步 main 分支
内容：
- 合并 main 分支并同步当前分支基线
- 保持合并提交标题和正文符合 Holmas 八位编号规则
```

编号选择规则：

- 如果本次 merge 的实际改动能稳定归到单一模块，使用该模块的下一个编号
- 如果本次 merge 横跨多个模块且无法稳定归类，使用通用模块编号，并在正文里说明同步范围
- `合并：` 是 merge 提交专用前缀；不要把默认英文 `Merge ...` 标题留进历史
- `.githooks/commit-msg` 会在提交前识别默认 `Merge ...` 标题，自动改写成编号标题和正文，再执行编号校验

## 确认词映射与执行口径

确认词固定映射为：

- `1`
  - 表示：提交并推送
- `2`
  - 表示：只提交
- `确认`
  - 表示：只提交
- `提交`
  - 表示：只提交
- `直接提交`
  - 表示：只提交

执行语义固定为：

1. `只提交`
- 优先复用 `.git/codex/pending_commit_suggestion.json` 中最近一次缓存的 `标题` 与 `内容`
- 使用该缓存执行本地 `git commit`

2. `提交并推送`
- 先按同样方式执行本地 `git commit`
- 再推送当前分支的上游远端

默认不写死：

- 不把 `main -> origin/main` 写成长期固定规则
- 长期口径统一写成“当前分支的上游远端”

## 缓存复用与执行前检查

最近一次“尚未提交、但已给出的提交建议”缓存到：

```text
.git/codex/pending_commit_suggestion.json
```

主线程确认执行前固定检查：

- 当前工作区里是否存在与本轮无关、且不应混入本次提交的脏改
- 当前暂存区或待暂存文件是否能清晰界定为“本轮改动”
- 缓存里的 `head_commit` 是否仍然对应当前仓库基线

如果存在下面任一情况，应暂停而不是盲目提交：

- 工作区里存在明显无关改动且无法安全排除
- 缓存已失效或不可用
- 当前 HEAD 已变化，无法确认缓存标题是否仍对应当前基线

相关辅助入口：

- `python3 tools/doc_maintenance/update_project_docs.py --doc-root doc show-pending-commit`
  - 只负责显示最近一次缓存的提交建议
  - 不负责自动消费确认词
- `python3 tools/doc_maintenance/update_project_docs.py --doc-root doc sync-commit-sequences --fetch`
  - 根据最新 git history 与远端基线刷新模块编号登记文件
- `python3 tools/doc_maintenance/update_project_docs.py --doc-root doc validate-commit-message --message-file <path>`
  - 按最新基线校验当前提交标题是否仍是该模块的正确下一个编号

## 默认生效方式

新会话只要读到这页，就默认视为：

- 已启用“按长期主文档规则执行”的状态
- 不需要用户再补一条固定口令
- 主线程必须自己判断当前该先 `briefing`，还是直接进入 `execution planning`

固定要求：

- 不要把“阶段判断”交还给用户
- 不要先追问“现在是不是要进入 execution dispatch”
- 只有用户显式限制“先只给建议”“这轮不要开真实 subagent”“只让主线程做”时，才覆盖默认自动判断

## 第一反应规则

### 情况 A：任务还不够明确

如果用户是在问“现在该做什么”“先看主线”“帮我判断下一步”，或当前目标还不够具体，先进入轻量 `briefing`。

固定读取：

- [项目总览](/Users/bruce/work/Holmas/doc/长期主文档/项目总览.md)
- [迭代记录索引](/Users/bruce/work/Holmas/doc/迭代记录/迭代记录索引.md)
- 最新迭代记录中的 `当前状态`、`关键结论`、`风险与阻塞`、`下一步`、`给下一轮的人`

固定输出：

- 当前主线
- 当前阻塞
- `1 ~ 3` 个任务建议
- 推荐先做哪个

默认不做：

- 不扫项目代码
- 不通读全量协作文档
- 不直接拉起真实 subagent

### 情况 B：任务已经明确

如果用户已经明确给出要改什么、查什么、评审什么、优化什么，主线程应直接进入 `execution planning`，不要再等用户补“开始执行”“跳过 briefing”之类的口令。

这里的“任务明确”至少满足下面两项：

- 目标对象明确
- 预期产出明确
- 是否属于代码 / 文档 / 审查 / 测试线足够判断

默认动作：

- 先用最短路径确认这轮到底要做什么
- 直接判断主线程直做、只启 helper，还是启动真实 subagent
- 在需要用户确认的大任务上，先给出执行方案，再进入实做

## 执行前方案格式

只要你确认开始执行，主线程给出的方案必须至少包含下面几项：

- 本轮目标
- 本轮归属职责
- 采用方式：`主线程直做` / `主线程 + helper` / `主线程 + 真实 subagent`
- 为什么这样判断
- 阶段里程碑和是否进入 Agent 6 审查闭环
- 收尾方式

其中“收尾方式”必须明确写成：

```text
按 /Users/bruce/work/Holmas/doc/长期主文档/协作与执行/任务完成后自动维护文档.md 中的规范收尾。
```

## subagent 判断规则

主线程必须自己判断“主 agent 一个就够”还是“值得开真实 subagent”，不要把这个决定丢回给用户。

默认判定：

- 满足下面情况时，优先 `主线程直做`
  - 单点修改
  - 高耦合改动
  - 需要边看边改、频繁来回验证
  - 真实 subagent 反而会增加协调成本
- 满足下面情况时，可考虑 `主线程 + helper`
  - 主要是只读梳理、搜集上下文、规则核对、日志整理
  - 主线程仍是唯一实现者
- 满足下面情况时，可考虑 `主线程 + 真实 subagent`
  - 可以拆成边界清晰的并行子任务
  - 写入目录可以独占
  - 审查、测试或专项实现可独立推进
  - 启动真实 subagent 明显快于主线程串行完成

固定要求：

- 不要为了“看起来专业”而机械开 subagent
- 方案里必须解释为什么这轮需要，或为什么这轮不需要
- 如果启了真实 subagent，必须写清每个 subagent 的职责、边界、交付物和汇报方式

## 真实 subagent 方案格式

如果这轮判断需要真实 subagent，方案里固定写清：

- subagent 名称或职责号
- 负责目标
- 必带 skill
- 允许写入范围
- 禁止写入范围
- 依赖前提
- 明确交付物
- 完成后汇报格式
- 是否需要进入 Agent 6 审查闭环

推荐写法：

```text
本轮采用：主线程 + 真实 subagent
判断理由：……

Subagent 1：
- 职责：……
- 必带 skill：……
- 允许写入：……
- 禁止写入：……
- 交付物：……

Subagent 2：
- 职责：……
- 必带 skill：……
- 允许写入：……
- 禁止写入：……
- 交付物：……

收尾：
按 /Users/bruce/work/Holmas/doc/长期主文档/协作与执行/任务完成后自动维护文档.md 中的规范收尾。
```

## 收尾硬门

只要这轮发生了 repo 级改动，结束前就必须进入收尾阶段。

固定要求：

- 不允许把 `append-iteration + sync` 视为已完成收尾
- 默认必须执行 `bash scripts/finalize_task.sh`
- 默认完整链固定为：`finalize_task.sh -> check-last-finalize -> final`
- `final` 消息里必须显式包含 `文档维护 / Git 提交建议 / 会话建议` 三段内容
- 发送 final 前，必须通过 `python3 scripts/update_project_docs.py --doc-root doc check-last-finalize`

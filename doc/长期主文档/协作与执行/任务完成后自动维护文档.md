## Summary

这份文档固定 Holmas 项目的任务收尾流程。

固定边界：

- 这页负责“什么时候必须收尾、收尾必须输出什么、`finalize_task.sh` / `check-last-finalize` 怎么用”
- Git 提交建议的具体格式、确认词映射和 `commit / push` 执行口径统一看 [Git 提交建议与确认规则](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Git%20提交建议与确认规则.md)

原则固定为：

- 代码完成，不等于任务完成
- 文档收尾完成，才算这一轮真的结束
- Git 提交前，必须能回答“这轮改了什么、风险是什么、下一步是什么”
- `方案与数据` 目录下的长期方案文档必须维护 `## 完成情况`；迭代记录负责写“这轮做了什么”，方案文档负责写“这个方案现在完成到哪了”
- 只要发生 repo 级改动，在 final 前必须执行 `finalize_task.sh` 或 `update_project_docs.py suggest-handoff`；单独执行 `append-iteration` 只算半收尾，视为未完成收尾
- 日常默认收尾入口固定为 `tools/doc_maintenance/finalize_task.sh`；不要再手工拼 `append-iteration + sync + 自己总结`

## 收尾工具

这套机制分成两层：

- `tools/doc_maintenance/finalize_task.sh`
  - 负责真正执行任务收尾
  - 追加迭代记录
  - 同步 `项目总览.md`、`主文档索引.md`、`迭代记录索引.md`
  - 自动暂存 `doc/` 下文档改动
  - 默认输出固定三段：`文档维护`、`Git 提交建议`、`会话建议`
  - 如果适合提交，默认生成中文 `标题：` 和 `内容：`
  - 如果适合提交，默认追加一条 `提交确认` 提示；提示词格式统一看 [Git 提交建议与确认规则](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Git%20提交建议与确认规则.md)
  - 完整收尾后会写入仓库 `.git/codex/last_finalize_report.json`

- `python3 tools/doc_maintenance/update_project_docs.py check-last-finalize`
  - 负责校验最近一次完整收尾状态是否仍然匹配当前仓库
  - 固定检查 `head_commit`、`worktree_status` 和固定三段收尾输出
  - 只有它返回 `[ok]`，才允许发送 final

- `python3 tools/doc_maintenance/update_project_docs.py append-iteration`
  - 只负责更新迭代记录
  - 不负责生成 `Git 提交建议`、`会话建议` 或 `迭代记录建议`
  - 执行后必须继续执行 `suggest-handoff` 或直接改用 `finalize_task.sh`
  - 如果只停在这一步，属于“半收尾”，不能视为任务结束
  - 执行后会使最近一次完整收尾状态失效

- `python3 tools/doc_maintenance/update_project_docs.py sync`
  - 只负责同步索引
  - 不负责生成收尾建议
  - 如果它产生了新的 repo 改动，同样会使最近一次完整收尾状态失效

- `python3 tools/doc_maintenance/update_project_docs.py suggest-current-commit`
  - 只负责基于当前仓库状态快速输出一次 `Git 提交建议`
  - 适合“用户只发了 Git 提交规则页，需要立即先给建议”的场景
  - 默认会复用短时远端基线缓存，并把最近一次建议写入 `.git/codex/pending_commit_suggestion.json`

- `.githooks/pre-commit`
  - 负责提交前兜底检查
  - 如果本次提交包含实现或流程相关改动，但没补迭代记录，就直接阻止提交
  - 提交前会自动同步索引，避免索引漏更新
- `.githooks/commit-msg`
  - 负责真正提交前的八位编号二次校验
  - 会先尽量 fetch 远端基线，再按最新 history 和模块编号登记文件校验标题
- `.githooks/post-commit`
  - 负责提交成功后的模块编号登记
  - 会同步 `协作与执行/状态登记/commit_module_sequences.json`，并把登记文件 amend 进刚刚生成的同一个提交

## finalize_task.sh 调用时机

固定在下面 3 个时机调用：

1. 完成一个明确里程碑后
2. 准备结束当前对话前
3. 准备 `git commit` 之前

不建议在下面场景调用：

- 纯问答、翻译、改一句文案
- 没有改变项目状态的事务性协助
- 任务做到一半，还没有形成可描述的小里程碑

补充规则：

- 如果同一轮 `summary / done / risk / next` 因参数修正重复执行，`append-iteration` 会跳过同一标题下已存在的相同条目，避免同一轮收尾被重复追加
- 参数修正仍应优先一次性补齐后再执行；幂等去重是兜底，不应替代正确填写 `--agent6-review`、`--session-mode`、`--iteration-mode`

## 完整收尾状态

`finalize_task.sh` 完成后，仓库会记录最近一次完整收尾状态到：

```text
.git/codex/last_finalize_report.json
```

这份状态至少用于回答 3 个问题：

- 最近一次完整收尾是什么时候发生的
- 当时输出了什么 `文档维护 / Git 提交建议 / 会话建议`
- 当前工作区是否还是对应那次收尾时的状态

固定约束：

- `append-iteration`、`new-iteration`、`sync` 这类会改项目文档的动作，都会使旧的完整收尾状态失效
- 如果当前有 repo 改动，但仓库里没有有效的最近一次完整收尾状态，就不应发送 final
- 需要人工核对时，可执行 `python3 tools/doc_maintenance/update_project_docs.py show-last-finalize`
- 发送 final 前，必须额外执行 `python3 tools/doc_maintenance/update_project_docs.py --doc-root doc check-last-finalize`

## 回复前固定流程

只要这轮发生了 repo 级改动，回复前固定按下面顺序执行：

1. `bash tools/doc_maintenance/finalize_task.sh ...`
2. `python3 tools/doc_maintenance/update_project_docs.py --doc-root doc check-last-finalize`
3. 只有在 `check-last-finalize` 返回 `[ok]` 后，才允许准备发送 final
4. 最终 `final` 消息里，必须显式包含 `文档维护 / Git 提交建议 / 会话建议` 三段内容；少任意一段都视为未完成收尾

如果第 2 步失败：

- 说明最近一次完整收尾状态已经失效，或根本没有完成收尾
- 此时不允许直接 final
- 必须重新执行 `finalize_task.sh`，或先查 `show-last-finalize` 再补收尾

如果第 4 步没做到：

- 即使 `check-last-finalize` 返回 `[ok]`，也不算真正完成收尾
- 不允许用“已执行收尾脚本”“已通过校验”“已完成文档维护”这些一句话代替固定三段输出
- 默认应直接复用 `finalize_task.sh` 刚生成的三段内容，不要在最终回复里自行删改成另一种结构

## 推荐调用方式

```bash
bash tools/doc_maintenance/finalize_task.sh \
  --summary "本轮完成了什么" \
  --done "已完成项" \
  --next "下一步" \
  --agent6-review passed
```

如果这轮确实改变了某个 Agent 的分工状态，再显式追加：

```bash
bash tools/doc_maintenance/finalize_task.sh \
  --summary "本轮完成了什么" \
  --agent-status "Agent 5：已启动并完成第 1 组与第 2 组核心边界测试补强"
```

如果这轮还有风险：

```bash
bash tools/doc_maintenance/finalize_task.sh \
  --summary "完成 Agent 2 地图与棋盘第一阶段" \
  --done "接入 terrain -> BoardTemplate" \
  --risk "揭示扩散规则还未验证" \
  --agent6-review pending \
  --next "开始接数字计算和通关判定"
```

如果这次不想自动清理临时验证工程，可追加：

```bash
bash tools/doc_maintenance/finalize_task.sh --summary "本轮完成了什么" --skip-temp-cleanup
```

## 收尾固定输出

每次任务结束时，固定至少输出三段：

```text
文档维护：已执行
Git 提交建议：适合提交
标题：xxxx
内容：
```text
- xxxx
- xxxx
```
会话建议：继续当前会话
```

如果不满足“适合提交”，则固定写成：

```text
Git 提交建议：暂不建议提交。原因是：xxxx
```

固定要求：

- 不允许省略 `文档维护`
- 不允许省略 `Git 提交建议`
- 不允许省略 `会话建议`
- 最终 `final` 消息必须显式包含这三段；脚本里生成过但最终没发给用户，仍视为未完成
- 如果 `Git 提交建议：适合提交`，必须继续补中文 `标题：` 和 `内容：`
- 如果 `Git 提交建议：适合提交`，必须继续补 `提交确认：...`
- `提交确认：...` 的固定提示词和确认词映射统一看 [Git 提交建议与确认规则](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Git%20提交建议与确认规则.md)
- `内容` 统一放进一个 `text` 代码块

## Git 提交建议与确认入口

这页只保留收尾阶段与 Git 提交建议相关的最小硬门：

- 如果收尾输出里出现 `Git 提交建议：适合提交`
- 就必须继续补 `标题：`
- 必须继续补 `内容：`
- 必须继续补 `提交确认：...`

具体规则统一看：

- [Git 提交建议与确认规则](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Git%20提交建议与确认规则.md)
- [Holmas 八位提交编号方案](/Users/bruce/work/Holmas/doc/长期主文档/方案与数据/Holmas%20八位提交编号方案.md)

固定补充规则：

- 即使这轮只是事务性协助，也必须给出 `Git 提交建议`
- `tools/doc_maintenance/finalize_task.sh` 默认会直接输出中文提交标题和内容
- 提交标题继续沿用正式八位编号方案
- 所有新建提交的最终标题都必须以 `[TMMSSSSS]` 八位编号开头，并通过 `.githooks/commit-msg` 的模块流水号校验
- 模块编号登记由 `.githooks/post-commit` 在提交成功后写入同一个提交，避免 `commit-msg` 阶段改 index 留到下一次提交
- 新建 merge 合并提交也必须走正式八位编号标题与正文规则；历史旧 merge / 旧无编号提交不回填、不占号
- `Revert`、`fixup!`、`squash!` 不再绕过编号规则；如需表达这些语义，也必须改写成正式编号标题
- 已经真正提交过的编号记录在 git history 里
- 最近一次“尚未提交、但已给出的提交建议”缓存到仓库 `.git/codex/pending_commit_suggestion.json`
- 如果脚本给出的提交建议在复杂多主线场景下不够精确，可以人工改写，但不能省略
- 主线程在给出 `Git 提交建议：适合提交` 后，必须主动追加一次提交确认，不要让用户再自己组织完整提交指令

```text
Agent 6 审查：已完成

## 会话建议与迭代记录建议

从现在开始，这两个判断固定拆开：

- `会话建议`
  - 只回答“当前是否建议切到一个新的 Codex 会话”
- `迭代记录建议`
  - 只回答“当前是否应该执行 `new-iteration`，切到新的迭代记录文件”

默认规则：

- `Agent 6` 未完成或未通过时：
  - 不生成下一个会话可复制提示词
  - 只允许 `继续当前会话` 或 `建议新开修复 / 复审会话`
- `Agent 6 = pending`
  - 表示审查已发起、结果未回
- `Agent 6 = deferred`
  - 表示当前处于 `待回补复审`
  - 允许继续当前修复 / 验证会话
  - 仍然不生成下一个会话可复制提示词
- 如果本会话已经出现过自动压缩背景信息，或当前会话已经连续完成两个以上大任务，默认倾向建议新开会话
- 只有 `Agent 6` 已通过，且下一步明显切到新主题时，才给 `建议新开下一阶段会话`

新增固定要求：

- 如果 `会话建议` 是任何一种 `建议新开...会话`，最终回复里必须继续补：
  - `下一个会话描述：...`
  - `下一个会话可复制提示词：`
  - 一个 `text` 代码块，里面放可直接复制到新会话的首条消息
- 如果 `会话建议` 不是新开会话，固定写：
  - `下一个会话描述：当前不生成，继续当前会话即可。`
  - `下一个会话可复制提示词：当前不生成，继续当前会话即可。`

## Git 提交流程

提交固定按这个顺序执行：

1. 完成代码或文档任务
2. 先执行 `bash tools/doc_maintenance/finalize_task.sh ...`
3. 再检查这轮迭代记录是否写清楚
4. 再 `git add` 代码改动
5. 最后才 `git commit`

## 提交前 hook 约束

首次启用本仓库的提交流程：

```bash
bash tools/repo_maintenance/install_git_hooks.sh
```

启用后，`.githooks/pre-commit` 会自动执行：

- 检查本次暂存区是否包含实现相关改动
- 自动同步两个索引文件
- 如果没有补迭代记录，则直接阻止提交

它的定位不是替你写总结，而是防止“代码改完就提交，文档完全没补”。

同时：

- `.githooks/commit-msg` 会强制校验八位编号标题
- `.githooks/post-commit` 会把最新模块编号登记 amend 进同一个提交

## 先判断是否值得写入文档

需要写入：

- 代码实现发生变化
- 配置、资源路径、脚本、测试、验证链路发生变化
- 长期规则、协作流程、文档维护机制发生变化
- 会影响后续接手人判断“做到哪了、风险是什么、下一步是什么”的结果

可以不写入：

- 纯事务性协助
- 只整理一句提交文案
- 不改变项目状态的临时沟通

即使不写项目文档，也仍然要给出固定三段收尾，只是 `文档维护` 会写成未执行原因。

## 推荐节奏

- 开始一轮正式工作前，先确认当前是否已有活跃迭代记录
- 完成一个明确里程碑后，立即执行一次 `bash tools/doc_maintenance/finalize_task.sh`
- 准备提交前，让 pre-commit 自动兜底检查

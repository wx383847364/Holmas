# 文档维护与 Git 提交流程

这页只负责任务收尾、Git 提交建议和 `finalize_task.sh` 的调用规则。

这页不重复解释：

- Agent 启动方式
- 线程调度阶段
- Agent 6 完整路由和职责边界

启动和审查链规则统一看：

- [Agent 启动与验收规范](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Agent 启动与验收规范.md)

## Summary

这份文档固定 Holmas 项目的任务收尾流程。

原则固定为：

- 代码完成，不等于任务完成
- 文档收尾完成，才算这一轮真的结束
- Git 提交前，必须能回答“这轮改了什么、风险是什么、下一步是什么”
- `方案与数据` 目录下的长期方案文档必须维护 `## 完成情况`；迭代记录负责写“这轮做了什么”，方案文档负责写“这个方案现在完成到哪了”
- 只要发生 repo 级改动，在 final 前必须执行 `finalize_task.sh` 或 `update_project_docs.py suggest-handoff`；单独执行 `append-iteration` 只算半收尾，视为未完成收尾

## 收尾工具

这套机制分成两层：

- `scripts/finalize_task.sh`
  - 负责真正执行任务收尾
  - 追加迭代记录
  - 同步 `项目总览.md`、`主文档索引.md`、`迭代记录索引.md`
  - 自动暂存 `doc/` 下文档改动
  - 默认输出固定三段：`文档维护`、`Git 提交建议`、`会话建议`
  - 如果适合提交，默认生成中文 `标题：` 和 `内容：`
  - 如果适合提交，默认追加一条 `提交确认` 提示，提醒用户可直接回复 `1 / 确认 / 提交 / 直接提交`

- `python3 scripts/update_project_docs.py append-iteration`
  - 只负责更新迭代记录
  - 不负责生成 `Git 提交建议`、`会话建议` 或 `迭代记录建议`
  - 执行后必须继续执行 `suggest-handoff` 或直接改用 `finalize_task.sh`
  - 如果只停在这一步，属于“半收尾”，不能视为任务结束

- `.githooks/pre-commit`
  - 负责提交前兜底检查
  - 如果本次提交包含实现或流程相关改动，但没补迭代记录，就直接阻止提交
  - 提交前会自动同步索引，避免索引漏更新

## finalize_task.sh 调用时机

固定在下面 3 个时机调用：

1. 完成一个明确里程碑后
2. 准备结束当前对话前
3. 准备 `git commit` 之前

不建议在下面场景调用：

- 纯问答、翻译、改一句文案
- 没有改变项目状态的事务性协助
- 任务做到一半，还没有形成可描述的小里程碑

## 推荐调用方式

```bash
bash scripts/finalize_task.sh \
  --summary "本轮完成了什么" \
  --done "已完成项" \
  --next "下一步" \
  --agent6-review passed
```

如果这轮确实改变了某个 Agent 的分工状态，再显式追加：

```bash
bash scripts/finalize_task.sh \
  --summary "本轮完成了什么" \
  --agent-status "Agent 5：已启动并完成第 1 组与第 2 组核心边界测试补强"
```

如果这轮还有风险：

```bash
bash scripts/finalize_task.sh \
  --summary "完成 Agent 2 地图与棋盘第一阶段" \
  --done "接入 terrain -> BoardTemplate" \
  --risk "揭示扩散规则还未验证" \
  --agent6-review pending \
  --next "开始接数字计算和通关判定"
```

如果这次不想自动清理临时验证工程，可追加：

```bash
bash scripts/finalize_task.sh --summary "本轮完成了什么" --skip-temp-cleanup
```

## 收尾固定输出

每次任务结束时，固定至少输出三段：

```text
文档维护：已执行
Git 提交建议：适合提交
标题：xxxx
内容：
```text
- xxxx
- xxxx
```
会话建议：继续当前会话
```

如果不满足“适合提交”，则固定写成：

```text
Git 提交建议：暂不建议提交。原因是：xxxx
```

固定要求：

- 不允许省略 `文档维护`
- 不允许省略 `Git 提交建议`
- 不允许省略 `会话建议`
- 如果 `Git 提交建议：适合提交`，必须继续补中文 `标题：` 和 `内容：`
- 如果 `Git 提交建议：适合提交`，必须继续补 `提交确认：如需我直接提交到 git，请回复 1 / 确认 / 提交 / 直接提交。`
- `内容` 统一放进一个 `text` 代码块

## Git 提交建议规则

只要当前改动已经形成边界清晰、验证完成、适合独立提交的里程碑，就写：

```text
Git 提交建议：适合提交
标题：xxxx
内容：
```text
- xxxx
- xxxx
```
```

如果当前改动仍混有多条主线、验证没跑完、审查没完成，或只是事务性协助，就写：

```text
Git 提交建议：暂不建议提交。原因是：xxxx
```

固定补充规则：

- 即使这轮只是事务性协助，也必须给出 `Git 提交建议`
- `scripts/finalize_task.sh` 默认会直接输出中文提交标题和内容
- 提交标题统一使用 `[23000001] 流程：……`、`[25000001] 文档：……` 这类格式
- 标题前缀按脚本分类规则自动选择：
  - 文档整理类：`文档：`
  - 流程 / 协作类：`流程：`
  - 审查 / 修复类：`修复：`
  - 测试 / 验证类：`测试：`
  - UI / 联调类：`UI：`
  - 配置 / 数值类：`配置：`
  - 玩法 / 地图 / 长期进度类：`玩法：`
  - 边界 / 骨架类：`架构：`
  - 无法识别时回退：`提交：`
- 提交编号固定使用 8 位：`[TMMSSSSS]`
- `T` 表示一级大类：`1=代码`、`2=文档`、`3=资源/美术/Unity资产`、`4=配置/数据`、`5=测试/验证`、`6=工具/生成器/项目脚本`
- `MM` 表示二级模块：例如 `230=协作与执行/收尾流程`、`250=策划配表文档`、`320=HotUpdate Res`、`620=UiPrefabGenerator`
- `SSSSS` 表示模块内递增流水号；同一模块从 `00001` 开始递增，不回填旧历史
- 当前冻结的模块表如下：
  - `110` `App.Shared / Contracts`
  - `120` `App.AOT / Bootstrap / Infrastructure`
  - `130` `App.HotUpdate / Holmas gameplay`
  - `140` `Minesweeper 接入`
  - `150` `运行时 UI 代码 / Presenter / Controller`
  - `210` `项目总览 / 主文档索引 / 入口页`
  - `220` `架构与边界`
  - `230` `协作与执行 / agent / skill / 收尾流程`
  - `240` `策划方案文档`
  - `250` `策划配表文档 / 数据方案文档`
  - `260` `迭代记录 / 启动卡 / 交接`
  - `270` `UI 自动生成系统专区文档`
  - `310` `图标 / 贴图 / 美术资源`
  - `320` `HotUpdate Res`
  - `330` `Scene / Prefab / Unity 资产`
  - `340` `地图 / Terrain / 关卡资源`
  - `410` `原始配置表 / CSV`
  - `420` `配置导出 / 转换脚本`
  - `430` `json / bytes / catalog 产物`
  - `440` `配置协议 / schema / 数据结构`
  - `510` `单元 / 集成测试`
  - `520` `smoke / validation / 回归`
  - `530` `边界检查 / QA 脚本`
  - `610` `通用项目脚本`
  - `620` `UiPrefabGenerator`
  - `630` `Editor 工具 / 导入器 / 生成辅助`
- 只有走这套正式提交流程的编号标题才视为占号；merge 和旧的无编号提交不占号
- 已经真正提交过的编号记录在 git history 里
- 最近一次“尚未提交、但已给出的提交建议”缓存到仓库 `.git/codex/pending_commit_suggestion.json`
- 如果脚本给出的提交建议在复杂多主线场景下不够精确，可以人工改写，但不能省略
- 主线程在给出 `Git 提交建议：适合提交` 后，必须主动追加一次提交确认，不要让用户再自己组织完整提交指令

## 提交确认规则

当收尾输出里已经给出：

```text
Git 提交建议：适合提交
标题：xxxx
内容：
```text
- xxxx
```
提交确认：如需我直接提交到 git，请回复 1 / 确认 / 提交 / 直接提交。
```

后续规则固定为：

- 如果用户回复 `1`
- 或回复 `确认`
- 或回复 `提交`
- 或回复 `直接提交`
- 或回复其他明确表示“按当前提交建议直接提交”的短语

主线程就按刚刚给出的 `标题` 和 `内容` 直接执行本轮 `git commit`，不要再重复追问一轮“要不要提交”。

执行前固定检查：

- 当前工作区里是否存在与本轮无关、且不应混入本次提交的脏改
- 当前暂存区或待暂存文件是否能清晰界定为“本轮改动”
- 如果存在明显无关改动且无法安全排除，主线程应暂停并说明原因，而不是盲目提交

默认执行方式：

1. 使用刚刚给出的中文 `标题`
2. 使用刚刚给出的中文 `内容`
3. 优先复用 `.git/codex/pending_commit_suggestion.json` 里缓存的最近一次提交建议
4. 只提交本轮相关改动

如果当前并不适合提交，则固定写：

```text
提交确认：当前不建议直接提交；如需强制提交，请明确说明。
```

## 会话建议与迭代记录建议

从现在开始，这两个判断固定拆开：

- `会话建议`
  - 只回答“当前是否建议切到一个新的 Codex 会话”
- `迭代记录建议`
  - 只回答“当前是否应该执行 `new-iteration`，切到新的迭代记录文件”

默认规则：

- `Agent 6` 未完成或未通过时：
  - 不生成下一阶段启动卡
  - 只允许 `继续当前会话` 或 `建议新开修复 / 复审会话`
- `Agent 6 = pending`
  - 表示审查已发起、结果未回
- `Agent 6 = deferred`
  - 表示当前处于 `待回补复审`
  - 允许继续当前修复 / 验证会话
  - 仍然不生成下一阶段启动卡
- 如果本会话已经出现过自动压缩背景信息，或当前会话已经连续完成两个以上大任务，默认倾向建议新开会话
- 只有 `Agent 6` 已通过，且下一步明显切到新主题时，才给 `建议新开下一阶段会话`

## Git 提交流程

提交固定按这个顺序执行：

1. 完成代码或文档任务
2. 先执行 `bash scripts/finalize_task.sh ...`
3. 再检查这轮迭代记录是否写清楚
4. 再 `git add` 代码改动
5. 最后才 `git commit`

## 提交前 hook 约束

首次启用本仓库的提交流程：

```bash
bash scripts/install_git_hooks.sh
```

启用后，`.githooks/pre-commit` 会自动执行：

- 检查本次暂存区是否包含实现相关改动
- 自动同步两个索引文件
- 如果没有补迭代记录，则直接阻止提交

它的定位不是替你写总结，而是防止“代码改完就提交，文档完全没补”。

## 先判断是否值得写入文档

需要写入：

- 代码实现发生变化
- 配置、资源路径、脚本、测试、验证链路发生变化
- 长期规则、协作流程、文档维护机制发生变化
- 会影响后续接手人判断“做到哪了、风险是什么、下一步是什么”的结果

可以不写入：

- 纯事务性协助
- 只整理一句提交文案
- 不改变项目状态的临时沟通

即使不写项目文档，也仍然要给出固定三段收尾，只是 `文档维护` 会写成未执行原因。

## 推荐节奏

- 开始一轮正式工作前，先确认当前是否已有活跃迭代记录
- 完成一个明确里程碑后，立即执行一次 `bash scripts/finalize_task.sh`
- 准备提交前，让 pre-commit 自动兜底检查

# 文档维护与 Git 提交流程

## Summary

这份文档固定 Holmas 项目的文档维护流程，重点约束两件事：

- 任务做完以后，怎么收尾
- Git 提交之前，怎么确认文档已经维护到位

从现在开始，这个项目不再把文档维护当成“顺手补一下”，而把它当成正式交付的一部分。  
原则固定为：

- 代码完成，不等于任务完成
- 文档收尾完成，才算这一轮真的结束
- Git 提交前，必须能回答“这轮改了什么、风险是什么、下一步是什么”

这套机制分成两层：

- `scripts/finalize_task.sh`
  - 负责真正执行任务收尾
  - 先判断这次任务是否值得写入项目文档
  - 对明确里程碑自动追加迭代记录
  - 对明确里程碑自动刷新 `项目总览.md` 里的“当前阶段”入口三行
  - 自动同步主文档索引和迭代记录索引
  - 自动暂存 `doc/` 下的文档改动
  - 自动尝试清理 `/tmp` 或 `/private/tmp` 下历史遗留的 Holmas 临时验证工程

- `.githooks/pre-commit`
  - 负责提交前兜底检查
  - 如果本次提交包含实现或流程相关改动，但没补迭代记录，就直接阻止提交
  - 纯资源提交默认放行，避免误拦美术同学
  - 提交前会自动同步索引，避免索引漏更新

## 收尾流程

每次完成一个明确任务、一个阶段目标、或准备结束本轮对话时，固定执行下面这套收尾动作：

这里的补充规则固定为：

- 一个有意义工作轮次对应一条迭代记录，不是每个小任务都新建新的记录文件
- 但每次任务结束时，都必须明确汇报这次文档维护是否已执行
- 每次一个大任务完成后，都要顺手判断当前改动是否适合形成一次 Git 提交
- 固定汇报格式只有两种：
  - `文档维护：已执行`
  - `文档维护：未执行。原因是：xxxx`
- 只有未执行时，才补充原因；已执行时不解释原因

1. 先整理这轮结果

- 本轮完成了什么
- 本轮还有什么风险或阻塞
- 下一步最合理的起点是什么

2. 执行统一收尾脚本

推荐命令：

```bash
bash scripts/finalize_task.sh --summary "本轮完成了什么" --done "已完成项" --next "下一步"
```

如果这轮确实改变了某个 Agent 的分工状态，再显式追加：

```bash
bash scripts/finalize_task.sh \
  --summary "本轮完成了什么" \
  --agent-status "Agent 5：已启动并完成第 1 组与第 2 组核心边界测试补强"
```

这里的 `分工状态` 默认文案固定读取：

- [Agent 启动与验收规范.md](/Users/bruce/work/Holmas/doc/长期主文档/协作与执行/Agent%20启动与验收规范.md) 的 `## 迭代文档默认分工状态`

使用规则固定为：

- 没有显式 `--agent-status`，就保持长期默认分工状态
- 只有本轮某个 Agent 真的承担了职责或启动了对应工作，才覆写那一行
- “验证 Agent 2 的结果”不等于“Agent 2 本轮已启动”

如果你明确不想在这次收尾里自动清理临时验证工程，可追加：

```bash
bash scripts/finalize_task.sh --summary "本轮完成了什么" --skip-temp-cleanup
```

如果这轮还有风险：

```bash
bash scripts/finalize_task.sh \
  --summary "完成 Agent 2 地图与棋盘第一阶段" \
  --done "接入 terrain -> BoardTemplate" \
  --done "完成关卡快照基础结构" \
  --risk "揭示扩散规则还未验证" \
  --next "开始接数字计算和通关判定"
```

如果这轮还需要做团结 / Unity batchmode 验证，推荐统一使用：

```bash
bash scripts/run_holmas_validation.sh
```

这条命令会：

- 自动复制当前工程到 `/private/tmp/holmas_validation_xxxxxx`
- 在临时副本里执行 EditMode 验证与 core logic smoke test
- 全部通过后自动删除本次创建的临时工程
- 如果验证失败，则保留临时工程和日志，方便排查

如果还要清理历史遗留的 Hub 临时项目记录，可用：

```bash
bash scripts/clean_hub_temp_projects.sh --dry-run
```

确认输出里都是临时项目后，再去掉 `--dry-run` 正式执行。  
这条脚本只处理 `/private/tmp` 或 `/tmp` 下符合 Holmas 临时命名规则的验证副本，不会处理 `work/` 下的正式项目。

3. 收尾脚本会自动完成这些动作

- 追加最新迭代记录
- 刷新 `项目总览.md` 中的“当前阶段 / 当前主线 / 当前建议起点”
- 同步 `主文档索引.md`
- 同步 `迭代记录索引.md`
- 自动暂存 `doc/` 下的文档改动
- 自动尝试执行 `scripts/clean_hub_temp_projects.sh`，清理历史遗留的 Holmas 临时验证工程
- 如果检测到仓库 `skills/` 真源有改动，还会顺手执行 `scripts/sync_codex_skills.sh`

关于自动清理的规则：

- 只清理 `/private/tmp` 或 `/tmp` 下符合 Holmas 临时命名规则的验证副本
- 不会处理 `work/` 下的当前项目或其他正式工程
- 如果检测到 Unity Hub / 团结 Hub 仍在运行，仍会继续删除 `/tmp` 下的临时工程目录，但会跳过 Hub 最近项目记录清理
- 如果清理脚本返回失败，只会输出警告，不会中断收尾流程
- 如确实不想在本次收尾时执行自动清理，可传 `--skip-temp-cleanup`

关于 skill 自动同步的规则：

- 只在 Git 检测到 `doc/长期主文档/协作与执行/skills/` 有改动时才执行
- 普通代码任务、普通文档任务如果没动 skill 真源，会自动跳过
- 自动同步调用的是 `bash scripts/sync_codex_skills.sh`
- 如果同步失败，只会给出警告，不会中断文档维护流程

4. 收尾完成后，才进入提交阶段

如果这一步没做，就不应该直接进入 `git commit`

5. 一个大任务结束后，顺手判断是否适合一次提交

- 如果当前改动已经形成一个清晰、边界明确、适合独立提交的里程碑，就直接给出 Git 提交建议
- Git 提交建议默认直接附在任务收尾回复里，不需要等用户再单独追问
- 如果当前改动明显还混着多条主线、不适合一次提交，要明确说明“暂不建议现在提交”
- 这一步是“提交建议”，不是强制立即提交；是否真的提交仍由用户决定

## 先判断是否值得写入文档

不是每一句协助、每一个问答、每一次事务性帮忙都要写进 `项目总览` 或 `迭代记录`。
从现在开始，先按下面规则判断：

- 需要写入：
  - 代码实现发生变化
  - 配置、资源路径、脚本、测试、验证链路发生变化
  - 长期规则、协作流程、文档维护机制发生变化
  - 会影响后续接手人判断“做到哪了、风险是什么、下一步是什么”的结果

- 默认不写入：
  - 帮忙起 Git 提交标题或提交内容
  - 纯解释、纯问答、快捷键说明、工具使用说明
  - 单纯润色一句话、改写文案、翻译一段文本
  - 与 Holmas 项目主线无关的临时操作

- 默认行为：
  - `scripts/finalize_task.sh` 会先做一次轻量判断
  - 如果识别为“事务性协助”，会直接跳过 `项目总览` 与 `迭代记录` 写入
  - 如果确实要强制落文档，可以显式传 `--force-log`

一个简单判断口径：

- 如果这轮结束后，下一位接手人有必要知道“项目状态变了”，就应该写
- 如果这轮只是帮你表达、解释、整理，不改变项目状态，就不该写

## 何时新建迭代记录

迭代记录不是每完成一个小任务就新建一份。  
这个项目固定按下面规则判断：

- 同一天如果只是同一主线下继续推进，继续写当天最新那一份迭代记录
- 同一天如果开启了新的“大主题”，再创建新的迭代记录，例如从“地图与任务纯逻辑”切到“UI 联调”或“资源管线重构”
- 跨天默认新建一份新的迭代记录，不继续沿用前一天的文件

这里的“大主题”指的是：

- 主线目标发生明显切换
- 承接人需要把这轮和上一轮清楚分开阅读
- 风险、完成项、下一步已经不适合继续混在同一份记录里

对应操作：

- 继续写当前活跃迭代记录：使用 `bash scripts/finalize_task.sh ...`
- 明确开启新一轮：先执行 `python3 scripts/update_project_docs.py new-iteration --title "本轮标题" --goal "本轮目标"`，再继续用 `finalize_task.sh`

注意：

- `scripts/finalize_task.sh` 默认只会追加到最新一轮，不会自动帮你判断“今天是否应该开 002”
- 是否新建迭代记录，先按上面的规则判断，再决定是否执行 `new-iteration`
- 即使这次任务没有新建迭代记录，也仍然要在任务结束时汇报 `文档维护：已执行` 或 `文档维护：未执行。原因是：xxxx`

## 使用方式

首次在本仓库启用 hook：

```bash
bash scripts/install_git_hooks.sh
```

完成一轮任务后，执行：

```bash
bash scripts/finalize_task.sh --summary "本轮完成了什么" --done "已完成项" --next "下一步"
```

如果有风险或阻塞，也一起写进去：

```bash
bash scripts/finalize_task.sh \
  --summary "完成 Agent 2 地图与棋盘第一阶段" \
  --done "接入 terrain -> BoardTemplate" \
  --done "完成关卡快照基础结构" \
  --risk "揭示扩散规则还未验证" \
  --next "开始接数字计算和通关判定"
```

默认行为：

- 默认写入最新一轮迭代记录
- 默认会先判断这轮是否值得写入项目文档；事务性协助会自动跳过
- 默认自动刷新 `项目总览.md` 的“当前阶段”区块
- 默认同步两个索引文件
- 默认暂存 `doc/` 下的变化
- 默认不会自动创建新的迭代记录
- 默认会自动尝试清理历史遗留的 Holmas 临时验证工程；如需跳过可传 `--skip-temp-cleanup`

其中：

- `主文档索引.md` 属于完全自动维护文档
- `项目总览.md` 目前自动维护的是“当前阶段”区块
- `项目总览.md` 的其他长期内容，例如“当前范围 / 稳定边界 / 当前不要做的事 / 进入 Agent 4 前条件”，仍然需要在项目阶段或稳定规则变化时手动更新
- `scripts/run_holmas_validation.sh` 默认会在验证通过后自动删除本次创建的临时工程，避免 `/tmp` 下残留越来越多测试副本
- `scripts/clean_hub_temp_projects.sh` 会被 `scripts/finalize_task.sh` 自动尝试调用，用来清理历史临时验证副本与可识别的 Hub 临时记录；如果 Hub 正在运行，则只删临时目录、跳过 Hub 记录清理；如果要单独预览，仍建议先 `--dry-run`

如果要写指定迭代记录，可用：

```bash
bash scripts/finalize_task.sh \
  --file "doc/迭代记录/迭代记录_20260329_001.md" \
  --summary "补充这轮收尾说明"
```

如果这轮虽然看起来像事务性协助，但你明确要留痕，可用：

```bash
bash scripts/finalize_task.sh --summary "补充本轮规则变更" --force-log
```

## 任务结束时的固定汇报

每次任务结束时，除了说明任务结果，还必须补一行文档维护状态：

```text
文档维护：已执行
```

如果这次没有执行文档维护，则改为：

```text
文档维护：未执行。原因是：xxxx
```

例如：

```text
文档维护：未执行。原因是：这轮只是帮你整理 Git 提交标题和内容，属于事务性协助，没有改变项目状态，所以不写入项目总览或迭代记录。
```

## Git 提交流程

Git 提交固定按这个顺序执行：

1. 完成代码或文档任务
2. 先执行 `bash scripts/finalize_task.sh ...`
3. 再检查这轮迭代记录是否写清楚
4. 再 `git add` 代码改动
5. 最后才 `git commit`

## 大任务完成后的 Git 提交建议格式

每次一个大任务完成后，都要顺手判断这轮改动是否适合形成一次提交。

如果适合，就直接按下面格式给出：

提交标题：

`docs: 让迭代文档分工状态改为长期规则源驱动`

提交内容：

```text
- 让迭代记录中的分工状态默认读取 5-Agent 长期规则源，不再使用待补充模板
- 为 update_project_docs.py 和 finalize_task.sh 新增 --agent-status 显式覆写能力
- 新增 backfill-agent-status，用于回填 20260330 之后旧迭代记录的分工状态占位
```

固定要求：

- 一律先写 `提交标题：`
- 再写单行标题
- 再写 `提交内容：`
- 内容统一放进一个 `text` 代码块
- 提交内容使用扁平短 bullet，不写嵌套层级
- 默认优先总结“这次改动解决了什么”，而不是逐文件罗列

如果不适合一次提交，就直接明确说明：

```text
这轮改动暂不建议现在合成一次提交。
原因：当前同时混有两条以上主线，建议先拆分后再分别生成提交标题和内容。
```

首次启用本仓库的提交流程：

```bash
bash scripts/install_git_hooks.sh
```

启用后，当前仓库会把 `core.hooksPath` 指向 `.githooks`。  
之后每次提交前，`.githooks/pre-commit` 都会自动执行：

- 检查本次暂存区是否包含实现相关改动
- 自动同步两个索引文件
- 如果没有补迭代记录，则直接阻止提交

它的定位不是替你写总结，而是防止“代码改完就提交，文档完全没补”。

## 约束规则

- 只要本次提交包含这些实现或流程相关改动，就必须补迭代记录：
  - `scripts/`
  - `ProjectSettings/`
  - `Packages/`
  - `.githooks/`
  - `Assets/` 下这些高影响类型：
    - `.cs`
    - `.asmdef`
    - `.prefab`
    - `.unity`
    - `.controller`
    - `.overridecontroller`
    - `.playable`

- 美术和纯资源提交默认放行，例如：
  - `.png`
  - `.jpg`
  - `.tga`
  - `.psd`
  - `.wav`
  - `.mp3`
  - `.fbx`
  - `.obj`
  - `.mat`
  - `.ttf`

- 资源对应的 `.meta` 文件会跟随资源类型一起判断：
  - 如果对应的是纯资源，`.meta` 也放行
  - 如果对应的是高影响类型，`.meta` 也继续纳入检查

- 纯 `doc/` 改动不会强制要求再补一条新的迭代记录。
- 索引文件属于派生文档，提交前 hook 会自动同步并暂存。
- hook 不能替你编造“本轮做了什么”，所以摘要、完成项、风险、下一步仍然由收尾脚本显式输入。

## 对话结束时的汇报规则

每次我结束一轮对话时，都会明确告诉你文档维护状态。  
输出格式固定为：

- `文档维护：已执行`
- 或 `文档维护：未执行。原因是：xxxx`

如果是“已执行”，我会补充说明：

- 更新了哪份迭代记录
- 是否同步了索引

如果是“未执行”，必须直接使用下面这个固定格式：

```text
文档维护：未执行。原因是：xxxx
```

其中 `xxxx` 至少要说明：

- 为什么这轮没有执行
- 是因为任务被判定为事务性协助、与项目无关，还是因为当前还不满足执行条件
- 如果后续需要执行，还差什么

这样你不需要自己猜“这轮有没有补文档”。

## 推荐节奏

- 开始一轮正式工作前，先确认当前是否已有活跃迭代记录
- 完成一个明确里程碑后，立即执行一次 `bash scripts/finalize_task.sh`
- 准备提交前，让 pre-commit 自动兜底检查

这样做的效果是：

- 迭代记录始终跟着实现走，不容易事后回忆补写
- 索引始终是最新的
- 下一轮接手时，可以直接看文档知道做到哪了

## Assumptions

- 这套机制是“脚本主动执行 + hook 被动兜底”的组合，不依赖聊天中手动提醒。
- `core.hooksPath` 是本地 Git 配置，所以每个仓库克隆后至少要执行一次 `bash scripts/install_git_hooks.sh`。
- 这套机制默认基于当前项目已有的 `scripts/update_project_docs.py`，不另建第二套文档系统。
